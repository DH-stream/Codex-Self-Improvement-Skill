# Repository agent instructions

1. Read `CORE.md` first.
2. For skill or procedure changes, use `superpowers:writing-skills`.
3. For multi-step changes, use `superpowers:writing-plans`.
4. Treat `tests/baseline-observations.md` as RED evidence and update relevant pressure scenarios before behavior changes.
5. Keep `skills/codex-self-improvement/SKILL.md` compact; move detail to `references/`.
6. Never overwrite or delete learned evidence; append history and change relevance/status instead.
7. Public repository files may contain only universal, anonymized, quality-preserving knowledge and neutral private templates.
8. Personal taste and private evidence belong only under installed `PRIVATE_LOCATION`; never add them to this repository.
9. After any prompt changing skill, workflow, installer, test, or memory schema files, run the post-change procedure.
10. A user correction or blocker follow-up requires the correction retrospective.
11. One qualified universal improvement may update a dedicated branch and draft PR; never push directly to `main` or merge automatically.
12. Do not weaken quality gates to optimize tokens.
13. Update public `memory/UPDATE_LOG.md` for material engine, universal-memory, installer, or schema changes.
14. After actual memory/skill writes, report only changed filename(s) and any draft PR reference unless the user asks for the lesson.
